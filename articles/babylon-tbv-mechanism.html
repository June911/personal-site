Babylon Trustless Bitcoin Vault

BTC 不离主链，让 DeFi 合约决定它的去向

TBV = Bitcoin UTXO + 联合预签 + BitVM3 验证。把外链智能合约状态的 ZK 证明，通过 garbled circuit 翻译成 BTC 主链可验证的 hash 揭示，不需要托管方、不需要永久签名委员会。本文聚焦：参与角色 · 密码学机制 · 完整流程。

1四个密码学构件

TBV 不是单一原语，是 4 个构件的精密组合。Bitcoin 主链只验最简单的"hash 原像"，复杂逻辑被推到链下电路。

SNARK

Groth16 证明

给 DeFi 链上智能合约状态转移生成简洁可验证证明。包含 MPT 状态根、BLS pairing、合约约束。

位置链下生成，链下验证
Garbled Circuit

电路混淆

把"验证 SNARK 证明"这件事编码成"输入触发 binary label"的电路。~43GB/对手方，存放在云。

位置链下存储，挑战时链下执行
Lamport Signature

一次性签名

可被 Bitcoin Script 验证，用于把 GC 输入"提交"到 BTC 链。仅在挑战路径上链，约 100KB/笔。

位置挑战时上链
Bitcoin Script

主链验证

Hash lock + Time lock + Taproot 多花费路径。Bitcoin 只看 secret 是否被揭示，不验任何复杂逻辑。

位置BTC 主链

2参与角色

每个 vault 创建时就固定了"谁可以索取"和"谁可以挑战"。角色集合的设计直接决定该 vault 的 trust 边界。

P2P 借贷场景（最简模型）

Borrower

Bob — 借款人 / BTC 持有者

把 BTC 锁进 vault，去 DeFi 链借 stablecoin。设置自己的 garbled circuit 防止 Larry 恶意清算；可以亲自做自己的 challenger。

持有自己的 secret + Larry 的 GC
Lender / Liquidator

Larry — 出借方

提供 stablecoin 流动性。设置自己的 GC 防止 Bob 假还款；当 Bob 违约时凭 ZK proof 取走 BTC。

持有自己的 secret + Bob 的 GC

三类 Vault Keeper 对比

官方 Vault Keeper Definition：每个 vault 创建时指定一组 Vault Keepers。三类的核心差异是代表谁和范围多大，不是"是否能挑战"——前两类都既能 claim 也能 challenge。

类型代表Claim 权限Challenge 权限典型实例
Vault Provider 单个用户 该 vault 该 vault BTC 持有者本人 / 付费第三方运营商
Application-elected 整个应用 该应用所有 vault 该应用所有 vault Aave 治理选定的 liquidator 集合
Universal Challenger 全局 无 所有 vault / 所有应用 独立运营的全局监督池

真实协议集成（如 Aave）的其他角色

Vault Provider

代表单个用户的 Vault Keeper

用户亲选的单一 Vault Keeper，双重职责：① 代表用户发起 redemption claim + 提交 ZK proof；② 监督针对该 vault 的恶意 claim 并挑战。可由 BTC holder 自己担任，也可付费委托给第三方运营商。

权限claim + challenge（仅限自己服务的 vault）
收入BTC Vault Redemption Fee 分成（链上声明）
运营vaultd 软件 · 4 CPU/16GB/100GB · 双链 keys · permissionless 注册
Application-elected Keepers

代表整个应用的 Keeper 集合

由集成应用（如 Aave）治理选定，可对该应用相关的所有 vault提交 claim（清算时）或挑战其他 keeper 的恶意 claim。对应白皮书 §5.1 的 k-of-n liquidators。

权限claim + challenge · 应用范围
Universal Challenger

全局监督池

跨应用、跨 vault，只能 challenge 不能主动 claim。三类 keeper 中唯一的"纯监督"角色，激励来自挑战成功的罚没分成。

权限仅 challenge · 全局范围
Arbitrageur

无许可清算套利者

wBTC pool 集成场景下，任何人在一区块内原子借 wBTC → 还债 → 取 vault 控制权 → 再质押。承担 wBTC↔BTC 基差风险（≈0）。

权限无需许可 · permissionless
Large Lender

j-of-m 大额出借人

vault 创建时联合签名。有动机不让 BTC 被恶意提走。小额 lender 的安全性回退到至少一个 large lender 集合诚实。

权限setup 联签
Light Client / Smart Contract

EVM 端代理层

Deposit smart contract 通过 Bitcoin light client 验证 vault → 铸造内部记账 ERC20 (collBTC) 喂给借贷协议。

位置EVM 链上

3P2P 借贷流程（教学模型）

以 Bob 借款人 + Larry 出借方为例。Setup 一次性，运行期分 Happy Path 和 Challenge Path 两条。

Setup — vault 创建（双方/多方协议）

关键：Setup 不是各自独立完成，是参与方之间的交互协议——creation and exchange 都要发生。每方 garble 一个"防自己作弊"的电路，交给对手方持有；对手方做 cut-and-choose 互验后才接受。
各方 garble 一个"防自己作弊"的电路 + transaction graph

Bob garble "验证 Bob 还款 proof"的电路（输出 = Bob 的 secret label）；Larry 对称做"防 Larry 恶意清算"的电路。每方同时构造所有可能花费交易的图（Withdraw / Slashing / Liquidation / ...）。

交换 GC + transaction graph

Bob 把自己的 GC 交给 Larry（Larry 将来用它监督 Bob）；Larry 把自己的 GC 交给 Bob（Bob 将来用它监督 Larry）。每份 ~43GB，链下存储。

Cut-and-choose 互验

Bob 验证 Larry 给的电路确实"在 Larry 作弊时才解出 secret"，反之亦然。概率失败率 2⁻⁴⁰（~10⁻¹²，极小但非零）。

互相预签所有花费交易 + 锁定 BTC

双方对 transaction graph 中的每条路径预签。BTC 存入 vault UTXO。整个 setup 一次性完成，运行期不再交互。

Vault Provider 提交 confirmation 到 EVM 端

setup 成功后，Vault Provider 向 ETH 端 deposit smart contract 提交"can service this vault"的确认；该合约负责跟踪 vault 状态（来自官方 §2.2）。

多 keeper 场景（真实集成）：当 vault keeper 不止两人（Vault Provider + Application-elected + Universal Challengers），所有指定 keeper 之间跑 n-by-n 配对 setup——每对之间都要 garble + 交换 + 互验。这是 setup 重的核心原因，也是 vault 创建后参与方集合不能再变的根因。

Happy Path — Bob 还款 + Larry 不挑战

Bob 链下生成"已还款"ZK proof

EVM 借贷合约关仓 + 销毁 collBTC，Bob 拿到状态变更证明。

Bob 在 BTC 链发 Claim tx

不上传 SNARK proof，不上传 Lamport sig，只发交易声明意图。

Larry 不挑战 → timelock 通过

Bob 取回 BTC。整个过程没有 GC 介入，没有 Lamport 上链，成本仅 ~$2.66（3 笔 BTC tx）。

cost: ~$2.66 · BTC tx: 3 · GC: 不触发

Challenge Path — Larry 怀疑欺诈

Larry 在 BTC 链发 Challenge tx

触发挑战机制，Bob 必须证明自己还款是真的。

Bob 用 Lamport 签名上传 ZK proof + 公共输入

~100KB 数据上链，Bitcoin Script 收到 proof 输入。这是 BitVM3 vs BitVM2 的核心节省点（$93 vs $15K）。

cost: ~$93 · BTC tx: ~100KB · 节省 170× vs BitVM2
Larry 链下用 Bob 的 GC 计算 binary label

Larry 拿 Bob 的 Lamport 签名 + 自己存的 GC，链下跑一遍电路。电路输出二选一标签。

分支 A：proof 有效（Bob 诚实）

GC 输出"合法" label → 解锁 Withdraw tx → Bob 等 timelock 后取回 BTC。Larry 的 challenge 失败，他不能再挑战这个 vault。

分支 B：proof 无效（Bob 作弊）

GC 的 "invalid" output wire label 在 setup 时就被 garble 为 secret_B 本身（即 Slashing tx 所需的 hash preimage）。Larry 拿到 secret 后只做一个动作：广播预签的 Slashing tx（witness 包含 secret_B + 已预签的 Larry 签名）。

这一个 tx 一旦 confirm，自动产生两个不可分割的后果：

(b) 惩罚：Slashing tx 把 vault BTC 转到 Larry 地址。
(a) 拦截：vault UTXO 已被花掉 → Bob 的 Withdraw tx 引用同一 UTXO，因双花互斥永远无法上链。

3.5Slashing 的具体实现 — UTXO + Hash Lock

Larry 拿到 secret 之后"如何让 BTC 进自己钱包 + 同时阻止 Bob 提款"——这不靠两步动作，靠 Bitcoin UTXO 模型的双花互斥 + Taproot 多 spend path。

Vault UTXO 的 Taproot 结构

Spend Path解锁条件谁可以花
Withdraw Bob 签名 + challenge timelock 已过 + UTXO 未被先花 Bob
Slashing（罚 Bob） Larry 签名（setup 已预签）+ 提供 x 使得 hash(x) == H_secret_B Larry，但需要 secret_B
Symmetric Slashing（罚 Larry） Bob 签名 + 提供 secret_L 作为 hash preimage Bob，但需要 secret_L

两个关键机制

机制 1

Slashing tx 在 setup 时已预签

Larry 不需要在挑战时找 Bob 重签。Setup 阶段双方对所有 spend path 都互签了。Slashing tx 的所有 ECDSA/Schnorr 签名齐了——唯一缺的 witness 就是 secret_B 作为 hash preimage。

机制 2

GC output label 直接 = secret

Garbled circuit 的输出线有两个 label。Setup 阶段 garble 时：L_1（valid）= 随机串；L_0（invalid）= secret_B 本身。Larry 跑 GC 拿到的字节串就是 BTC Script 需要的 hash 原像，无需查表。

完整动作序列（挑战成功）

Larry 链下：用 Bob's GC + Lamport-signed proof 跑电路

得到 32+ 字节输出标签。如果 proof 无效 → 标签 = secret_B。

Larry 立即广播预签的 Slashing tx

tx witness = { secret_B, Larry 签名 }。BTC 主网验证 hash(secret_B) == H_secret_B ✓ → tx 有效 → 等 confirm。

Slashing tx confirm — 一个动作，两个后果
后果 (b)：惩罚

Slashing tx 的 output 把 vault BTC 转到 Larry 地址。

后果 (a)：拦截

vault UTXO 已被花掉 → Bob 的 Withdraw tx 引用同一 UTXO，因双花互斥永远无法上链。Bob 即使等到 timelock 也取不到。

为什么这个设计精妙：Bitcoin Script 没有"NOT 揭示"运算符，无法直接表达"如果对手揭示了 secret 就废掉我的 Withdraw"。但 UTXO 模型天然提供——Slashing tx 一旦消费了 vault UTXO，引用同一 UTXO 的 Withdraw tx 自动作废。所以"拦截"不是单独编程出来的，是 Bitcoin 状态机内禀属性。

4Garbled Circuit 输出方向

Whitepaper 与 Blog 两份文档对"GC 何时输出 secret"的描述看似相反——其实是同一 binary label 机制的两个视角。

ZK Proof Bob 上传的"已还款"证明
→
Garbled Circuit 计算"proof 是否合法"
→
Binary Label L₀ 或 L₁
L₁（合法） Blog 视角：proof 有效 → 解锁 Withdraw → Bob 取 BTC（"reveals secret label if proof is valid"）
L₀（非法） Whitepaper 视角：proof 无效 → 解锁 Slashing → Larry 拿 BTC（"extraction only possible if proof is invalid"）
关键点：每对参与方各持有两个对称的 GC（Bob 的 GC 防 Bob 作弊，Larry 的 GC 防 Larry 作弊）。同一 secret 同时承担"否决欺诈提款"与"启动惩罚清算"两个角色。

5真实协议集成（Aave 等）

P2P 是教学例子。集成到 pool-based 借贷协议时，参与方扩展为 borrower + 多 liquidator + 多 lender。同时存在两条清算路径。

5.1 抵押 + 借款

借款人在 BTC 创建 vault，存入 BTC
把 vault 元数据（TXID / vault ID / 目标合约 / 借款金额）发给 EVM Deposit Contract
Deposit Contract 通过 Bitcoin light client 验证 → 铸造 collBTC
collBTC 存入借贷协议作为抵押 → 开仓借出 stablecoin

5.2 通用清算路径（白皮书 §5）

BTC 价格跌破阈值 → 头寸进入清算
白名单 liquidator 监控价格 + vault 状态
Liquidator 替借款人偿还债务 → 拿到 collBTC → 销毁
生成销毁 ZK proof，提交到 liquidator 自己的 BTC vault
过 timeout（几小时-2 天） → liquidator 取走 BTC 作为补偿
三大缺陷：① 无法部分清算（vault 是整体的） ② 清算人池受限（必须预设白名单） ③ 资金前置 + 价格风险（Liquidator 付了 USDC 还要等几小时-2 天才拿到 BTC）

5.3 wBTC pool 双池 + Arbitrageur 设计（Blog 优化版）

解决前两项缺陷；资金占用问题转化为 wBTC↔BTC 基差风险（≈ 0）。

Arbitrageur 从 wBTC 池借 wBTC（闪贷式）

无需许可，任何人都可执行。

卖出 wBTC 换 USDC（即时）
用 USDC 还原贷款，获得对应 vault 的 controllership

这一步把 vault 的 claim 权从原借款人手中转出。

把 vault 抵押给 wBTC 池

4 步在同一区块内原子完成。之后 wBTC 池的 keeper（在 vault claimer 集合内）才慢慢从 vault 提 native BTC、卖成 wBTC 还回池子。

结构性观察：该设计并不消灭 wBTC 池——反而短期依赖 wBTC 池作为清算缓冲层。前提是目标链已有大量低利用率闲置 wBTC（Aave 上长期符合）。长期看 wBTC 与 TBV 是共生而非纯替代。

7成本与摊销

BitVM3 vs BitVM2 的核心节省点：把链上 SNARK 验证压缩成"hash 揭示"。

场景数量成本vs BitVM2
Happy Path（无挑战）3 笔 BTC tx (Deposit/Claim/Withdraw)~$2.66—
Challenge case~100KB Lamport 签名 + 公共输入~$93170× 节省
